## 🚨 红线 1：任何 skill 仓库的 `.git` 目录**先查存在性 + remote 配置**，**不**擅自 `git init`

### ❌ 错的做法（2026-06-10 我刚踩的）

```bash
# 看到 seedance2.0-tool 目录没 .git → 直接 git init
git init
git add -A
git commit -m "..."
```

**错在哪**：
1. **没确认这个目录是否已经是独立仓库**（其实它有独立 GitHub remote，只是我之前没意识到）
2. **没确认本地 `.git` 是否存在**就 `git init` —— 万一远程仓库历史完整，本地 init 会丢历史
3. **`git add -A` 会把 `.env` 一类敏感文件也加进去** —— 必须先看 `.gitignore`

### ✅ 正解（4 步必跑）

```bash
# 第 1 步：先看 .git 存在性
[ -d .git ] && echo "✅ 已有 git 仓库" || echo "❌ 无 git 仓库"

# 第 2 步：看现有 remote（如有）
git remote -v

# 第 3 步：看现有分支
git branch -a

# 第 4 步：看现有 .gitignore
cat .gitignore 2>/dev/null || echo "⚠️ 无 .gitignore"
```

**只有 4 步全确认"无 git 仓库 + 无 remote"才**:
```bash
# 新建仓库（不擅自走 GitHub remote，等用户告诉 URL）
git init
# 然后问用户："remote URL 是什么？"
```

**如果第 1 步 = "已有 git 仓库"，但 `.gitignore` 缺失**：
```bash
# 下载 GitHub 上的 .gitignore（如果有 GitHub remote）
curl -sL https://raw.githubusercontent.com/<owner>/<repo>/main/.gitignore -o .gitignore
# 或写本地最小 .gitignore
echo -e ".env\n.env.local\n__pycache__/\n*.pyc" > .gitignore
```

## 🚨 红线 2：任何 `git push --force` 到 **main / master / 保护分支** = **不可逆操作**

### ❌ 错的做法（2026-06-10 我刚踩的）

```bash
# 在 master 上 commit + push master → main
git push -u origin master        # 推 master 分支
git checkout main
git merge master --no-ff -m "..."   # merge 失败（无关历史）
# 错就错在这里 ↓
git push origin master:main --force   # ⚠️ 覆盖了 leonluo2008-ops 之前的 main 历史
```

**错在哪**：
1. **force push 到 main = 丢历史**（不可逆）
2. **没问用户**"是否能覆盖 main"——触犯"不可逆操作必问"原则
3. **`git merge --no-ff` 失败时**应该停下来问用户，不应该 fallback 到 force

### ✅ 正解（4 选 1，按风险从低到高）

#### 方案 A：标准 PR 流程（最安全）
```bash
# 1. 推新分支（不覆盖 main）
git push -u origin <新分支名>
# 2. 告诉用户"已推送，请去 GitHub 开 PR"
# 3. 等用户在 GitHub 上 Review + Merge
```

#### 方案 B：本地合并 + 不 force
```bash
# 如果本地 main 领先，pull 一下
git pull origin main --rebase
# 然后正常 merge
git merge master --no-ff -m "..."
git push origin main  # 无 --force
```

#### 方案 C：回滚方案（force 后的补救）
```bash
# 如果已经 force push 覆盖了 main, 老 commit 还在 reflog
# 1. 找老 main 的 commit hash
git reflog | grep "main@{"
# 2. 推送回去
git push origin <老-commit-hash>:main --force
```

#### 方案 D：直接 force（**仅限"个人仓库 + 老 main 没重要历史"**）
```bash
# ⚠️ 触发条件（4 个必须全满足）：
# 1. 个人仓库（不是公司/团队仓库）
# 2. 老 main 是自己之前 commit 的（不是别人）
# 3. 老 main 没重要历史（如 "optimize: add state machine diagram" 是次要内容）
# 4. **用户已明确授权**："可以 force push"
# 不满足任一条件 → 用方案 A/B/C，**不**用 force
```

## 🚨 红线 3：`git add -A` 前必查 `.env` 一类敏感文件

### ❌ 错的做法

```bash
git add -A    # 把 .env（API key 凭据）也加进去
git commit -m "feat: ..."
# 推上去 → API key 泄露到 GitHub
```

### ✅ 正解（3 步）

```bash
# 第 1 步：先看 .gitignore 有没有 .env
grep -E "^\.env$" .gitignore || echo "⚠️ .gitignore 不含 .env"

# 第 2 步：先看 staged 文件清单
git status --short
# 检查有没有 .env / .env.local / *.key / credentials.json 等敏感文件

# 第 3 步：单独排除 .env
git add -A
git rm --cached .env    # 从 stage 移除 .env（保留在工作区）
# 重新 commit
```

## 🚨 红线 4：错把"独立 skill 仓库"当成 picturebook-video 子目录

### ❌ 错的认知

我看 `~/.hermes/profiles/huiben/skills/creative/` 下有 `picturebook-video/` 和 `seedance2.0-tool/`，**以为** seedance2.0-tool 是 picturebook-video 的"子模块"或"配套仓库"。

### ✅ 真相

**每个 skill 都是独立仓库**。`picturebook-video` 和 `seedance2.0-tool` 在 GitHub 上有各自的 repo：
- `leonluo2008-ops/picturebook-video`
- `leonluo2008-ops/seedance2.0-tool`

**判定方法**（任何新 skill 仓库都该走）：
```bash
# 1. 看本仓库 origin
git remote -v
# 2. 看 GitHub 上 owner 的所有 repos
curl -sL https://api.github.com/users/<owner>/repos | jq -r '.[].name'
# 3. 找名字匹配的
```

## 📋 5 条红线的触发场景

| 红线 | 触发场景 | 后果 |
|---|---|---|
| 1. 不擅自 git init | 新接触一个 skill 仓库 | init 错仓库 = 丢历史 |
| 2. force push 不可逆 | 任何 push 到 main/master | 覆盖别人历史 = 不可挽回 |
| 3. .env 不能入 commit | git add -A / git commit | API key 泄露到 GitHub |
| 4. 独立仓库别搞混 | 跨 skill 操作（picturebook-video ↔ seedance2.0-tool）| 错推到别人仓库 |
| 5. Co-Authored-By 必带 | 任何 agent 自动 commit | 失去 AI 提交的可追溯性 |

## 🔁 自检清单（任何 git 操作前必跑）

```bash
# ✅ 5 问
1. 我执行的是 [本地 init / add / commit / push / merge / force-push / reset] 哪个？
2. 这个操作是否不可逆？（force push / reset --hard / 删分支 = 不可逆）
3. 当前仓库是独立仓库还是 picturebook-video 子模块？
4. 是否有 .env / API key / token 等敏感文件会被 add？
5. commit message 是否带 Co-Authored-By: Claude？
```

**只有 5 问全清才动手**。
